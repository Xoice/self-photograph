# 云服务器部署指南

> 供 AI 代理（或人工）在云服务器上执行。按顺序逐步完成。

---

## 服务器要求

- **系统**：Linux（Ubuntu 22.04 推荐）
- **配置**：2 核 2G，40GB 硬盘
- **已装软件**：Node.js >= 20、npm、nginx、pm2
- **项目传输方式**：本地拖拽整个项目文件夹到服务器

---

## 第零步：配置国内镜像源（加速下载）

> 云服务器默认走国外源，npm install / apt install 会非常慢。先换国内源。

### npm 淘宝镜像

```bash
npm config set registry https://registry.npmmirror.com
```

### apt 清华镜像（Ubuntu 22.04）

```bash
# 备份原文件
sudo cp /etc/apt/sources.list /etc/apt/sources.list.bak

# 替换为清华源
sudo tee /etc/apt/sources.list << 'EOF'
deb https://mirrors.tuna.tsinghua.edu.cn/ubuntu/ jammy main restricted universe multiverse
deb https://mirrors.tuna.tsinghua.edu.cn/ubuntu/ jammy-updates main restricted universe multiverse
deb https://mirrors.tuna.tsinghua.edu.cn/ubuntu/ jammy-backports main restricted universe multiverse
deb https://mirrors.tuna.tsinghua.edu.cn/ubuntu/ jammy-security main restricted universe multiverse
EOF

sudo apt update
```

### 安装 Node.js 20（用 NodeSource，不走 apt 自带的旧版）

```bash
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt install -y nodejs
```

> 如果 NodeSource 也慢，可用 fnm（Rust 写的版本管理器，国内 CDN 加速）：
> ```bash
> curl -fsSL https://fnm.vercel.app/install | bash
> source ~/.bashrc
> fnm install 20
> fnm use 20
> ```

### 安装 nginx + pm2

```bash
sudo apt install -y nginx
npm install -g pm2
```

## 前置：服务器上构建（需加 swap 防止 OOM）

> **⚠️ 2 核 2G 服务器直接 `npm run build` 会 OOM 卡死。**
>
> 前端构建（rolldown-vite）需编译 11944 个模块，峰值内存 ~1.5GB。
> 必须先加 swap + 限制 Node 内存，否则进程被杀。
>
> **另外：Vite 构建会把 `.env` 的值写死进 JS 文件，所以构建前必须确保 `.env` 是生产地址！**

### 构建前先改好 .env

**根目录 `.env`**（前端，构建时写死进 JS）：

```env
VITE_API_BASE_URL=http://你的服务器IP/api/v1
VITE_UPLOAD_BASE_URL=http://你的服务器IP
VITE_BILIBILI_SPACE_URL=https://space.bilibili.com/你的B站ID
VITE_USE_MOCK=false
```

**`apps/api/.env`**（后端，运行时读取，不写死）：

```env
NODE_ENV=production
PORT=3000
DATABASE_URL=file:./dev.db
JWT_SECRET=换成一个强随机字符串（建议32位以上）
JWT_EXPIRES_IN=7d
CORS_ORIGIN=http://你的服务器IP
```

> **注意**：JWT_SECRET 必须修改，生产环境使用默认值会导致启动失败（main.ts 有检查）。
> 生成随机密钥：`node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"`

### 服务器上构建步骤

```bash
# 1. 加 2G swap（防止 OOM）
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile

# 2. 确认 .env 已改成生产地址
cat /www/self_photograph/.env
# 必须看到 VITE_API_BASE_URL=http://你的服务器IP/api/v1

# 3. 限制 Node 内存后构建前端（预计 2-5 分钟，慢但不会崩）
cd /www/self_photograph
NODE_OPTIONS="--max-old-space-size=1024" npm run build

# 4. 后端构建（轻量，几十个文件，不用限制）
cd apps/api
npm run build
npm run prisma:generate

# 5. 构建完删 swap（可选）
sudo swapoff /swapfile
sudo rm /swapfile
```

## 第一步：上传项目到服务器

将本地项目文件夹拖拽到服务器，建议放到 `/www/self_photograph`。

**不要上传的目录**（体积大或无用）：

| 目录 | 原因 |
|------|------|
| `.git/` | 不需要版本历史 |
| `apps/api/prisma/dev.db` | 生产环境重新建库 |

> `node_modules/` 是否上传取决于第二步选哪种方案：
> - 方案 A（本地传）：要上传
> - 方案 B/C（服务器装）：不上传

**必须上传的**：

- `dist/`（前端构建产物）
- `apps/api/dist/`（后端构建产物）
- `apps/api/prisma/`（schema + migrations + seed）
- `src/`、`apps/api/src/`（源码，部分配置可能需要）
- `package.json`、`package-lock.json`
- `apps/api/package.json`、`apps/api/package-lock.json`
- `.env`、`apps/api/.env`（生产配置）
- `public/`（robots.txt、sitemap.xml）
- `docker-compose.yml`（如需 PostgreSQL 备用）

## 第二步：安装依赖

> **⚠️ 2 核 2G 服务器 `npm install` 可能 OOM 崩溃。**
>
> 有三种方案，按推荐程度排序：

### 方案 A：本地传 node_modules（推荐，最省心）

前提：本地和服务器的 Node.js 大版本一致（都是 20）。

```bash
# 本地：压缩依赖（约 200-300MB）
cd E:\code\self_photograph
tar -czf deps-bundle.tar.gz node_modules apps/api/node_modules

# 拖到服务器后解压
cd /www/self_photograph
tar -xzf deps-bundle.tar.gz
```

跳过下面的 npm install。

### 方案 B：加 swap 后 npm install

```bash
# 1. 加 2G swap
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile

# 2. 配置淘宝镜像加速
npm config set registry https://registry.npmmirror.com

# 3. 装依赖
cd /www/self_photograph
npm install
cd apps/api
npm install

# 4. 装完后可删 swap（可选）
sudo swapoff /swapfile
sudo rm /swapfile
```

### 方案 C：限制 Node 内存

如果加了 swap 还是崩，限制 Node 内存上限：

```bash
NODE_OPTIONS="--max-old-space-size=1024" npm install
```

## 第五步：配置 nginx

创建 nginx 配置文件：

```bash
sudo nano /etc/nginx/conf.d/xoice.conf
```

写入以下内容：

```nginx
server {
    listen 80;
    server_name 你的服务器IP;

    # 前端静态文件
    root /www/self_photograph/dist;
    index index.html;

    # 前端路由（SPA history 模式）
    location / {
        try_files $uri $uri/ /index.html;
    }

    # API 反代
    location /api/ {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # 上传文件反代
    location /uploads/ {
        proxy_pass http://127.0.0.1:3000;
    }

    # 上传文件大小限制
    client_max_body_size 10m;

    # 静态资源缓存
    location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff|woff2|ttf|eot)$ {
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    # gzip 压缩
    gzip on;
    gzip_types text/plain text/css application/json application/javascript text/xml application/xml application/xml+rss text/javascript;
    gzip_min_length 1000;
}
```

测试并重载 nginx：

```bash
sudo nginx -t
sudo nginx -s reload
```

## 第八步：防火墙配置

> **⚠️ 严重警告：UFW 启用前必须先放行 SSH（22 端口），否则会立即失去服务器连接！**
>
> UFW 默认策略是 `deny all`（拒绝所有入站）。如果直接 `ufw enable` 而没有放行 22，SSH 会被拒，
> 你将无法再通过 SSH 连接服务器，只能通过云服务商控制台（如阿里云 VNC）恢复。
>
> **正确顺序：先 allow 22，再 allow 80，最后 enable。**

```bash
# 1. 先放行 SSH（必须！）
sudo ufw allow 22/tcp

# 2. 放行 HTTP
sudo ufw allow 80/tcp

# 3. 确认 3000 端口不对外开放
sudo ufw deny 3000/tcp

# 4. 最后才启用防火墙
sudo ufw enable

# 5. 验证规则
sudo ufw status
# 应看到：22 ALLOW、80 ALLOW、3000 DENY
```

如果已经锁死自己（SSH 连不上）：
1. 登录阿里云控制台 -> 实例 -> 远程连接 -> VNC
2. 执行 `sudo ufw allow 22/tcp` 或 `sudo ufw disable`

## 常见踩坑记录

> 以下为实际部署中遇到的问题，供 AI 代理或人工参考，避免重蹈覆辙。

### 1. UFW 防火墙锁死 SSH（严重）

**现象**：执行 `ufw enable` 后 SSH 断开，无法再连接服务器。

**根因**：UFW 默认 `deny all`，若未先 `allow 22` 就启用防火墙，SSH 端口被拒。

**预防**：启用前必须先执行 `sudo ufw allow 22/tcp`。

**恢复**：通过阿里云控制台 -> 远程连接 -> VNC 登录，执行 `sudo ufw allow 22/tcp` 或 `sudo ufw disable`。

### 2. npm install 超时 / 极慢

**现象**：`npm install` 卡住或耗时 30 分钟以上。

**根因**：云服务器默认走 npm 官方源（国外），国内访问极慢。

**解决**：见"第零步"配置淘宝镜像 `npm config set registry https://registry.npmmirror.com`。

### 3. prisma generate 报错 EPERM

**现象**：`prisma generate` 报 `EPERM: operation not permitted, rename ...query_engine-windows.dll.node`。

**根因**：Node.js 进程仍在运行（如 pm2 或 nest start --watch），占用了 Prisma 引擎文件。

**解决**：先 `pm2 stop xoice-api` 或关掉后端进程，再执行 `prisma generate`。

### 4. 中文文件名乱码

**现象**：从 Windows 拖拽到 Linux 服务器后，中文文件名变成 `╬╩╠т.md` 之类乱码。

**根因**：Windows 中文文件名用 GBK 编码，Linux 用 UTF-8，编码不匹配。

**解决**：上传后手动重命名 `mv 乱码文件名 正确文件名`，或本地改为英文文件名再上传。

### 5. pm2 restart 后代码未更新

**现象**：更新了 `apps/api/dist/` 后执行 `pm2 restart`，但接口行为没变。

**根因**：pm2 可能缓存了旧模块，或文件未正确覆盖。

**解决**：`pm2 delete xoice-api && pm2 start dist/main.js --name xoice-api`（彻底重启而非 reload）。

### 6. 前端能访问但 API 404

**现象**：首页能打开，但登录/接口请求返回 404。

**根因**：nginx 的 `location /api/` 反代配置缺失或 `proxy_pass` 地址错误。

**解决**：检查 nginx 配置中 `proxy_pass http://127.0.0.1:3000` 是否正确，`nginx -t && nginx -s reload`。

### 7. 服务器上 npm run build 卡死 / OOM 被杀

**现象**：在 2 核 2G 服务器上执行 `npm run build`，进程卡住不动或被 kill。

**根因**：前端构建（rolldown-vite）需编译 11944 个模块，峰值内存 ~1.5GB，2G 服务器内存不足，触发 OOM Killer 杀进程。

**解决**：不要在服务器上构建。本地 build 完只传 `dist/` 和 `apps/api/dist/` 到服务器。后端 `nest build` 很轻量（几十个文件），可以在服务器上跑。

**如果非要在服务器构建**（不推荐）：先加 swap：
```bash
fallocate -l 2G /swapfile
chmod 600 /swapfile
mkswap /swapfile
swapon /swapfile
```

### 8. 服务器上 npm install 也 OOM 崩溃

**现象**：`npm install` 执行到一半被 kill，或卡住不动。

**根因**：和构建一样，2G 内存不够 npm install 的峰值内存消耗。

**解决方案 A（推荐）**：本地 `node_modules` 压缩传上去，服务器跳过 install。
```bash
# 本地
tar -czf deps-bundle.tar.gz node_modules apps/api/node_modules
# 服务器解压
tar -xzf deps-bundle.tar.gz
```

**解决方案 B**：加 2G swap 后再 install（见第二步方案 B）。

**解决方案 C**：限制 Node 内存：`NODE_OPTIONS="--max-old-space-size=1024" npm install`。
